Locking/Mutex
  • Locking/Mutex and semaphore were the methods used initialy(olden times), that are used to manage mulitple threads.
Process involved in Locking/Mutex
  • In Locking/Mutex, Mutex is a variable that is equal to zero(Mutex = 0) which is assigned to a file initially and it means the file can be accessed by threads.
  • consider two threads, when thread1 starts to access it , it will lock the file by changing Mutex = 1.
  • Even when the time slice(alloted by the CPU) is gone for thread1, thread2 canâ€™t access it, since it is locked by thread1.
  • after thread2 time slice is gone, thread1 will come again and access it and untill thread1 finishes using the file thread2 canâ€™t do anything in itâ€™s time slice.
  • once thread1 finishes using the file it will make the Mutex=0.
  • now the same process repeats for thread2.
DrawBacks in Locking/Mutex
  • consider two threads has to access a file
  • Thread1 will see the Mutex = 0 and by the time it changes the Mutex to 1, the time slice is already gone.
  • now thread2 will see the Mutex = 0 and it will change it to 1 (say this thread has a little extra time).so thread2 saw it is free and locked it.
  • Thread1 time slice will come again, but thread1 already saw that file is free(2nd point), so it doesnâ€™t need to see again, Mutex is already 1 and Thread1 will also make it 1 and starts using it.
  • so both the thread will start using the file and print an unwanted ouput.

  • when the time slice is over for Thread1, Thread2 will see that the mutex is 1, so insted of wasting the CPU time(like Locking/Mutex) it will send that thread to blocked queue.
  • when time slize is over for thread 2, the same will happen for Thread3 and Thread 4 doesnâ€™t want to use the cpu time yet.
  • now thread1 is finished using the file, so it will call the signal method and the signal method will make the Mutex = 0.
  • then thread2 will start using it and it will come back to ready queue.
  • now thread2 will make the mutex = 1 and start using it, untill it finishes the other threads may be in blocked queue or ready queue.
  • once thread2 finishes file it will call the signal method and the signal method will make the mutex = 0.
  • same process will happen for Thread3 and Thread4.

Inter-Thread Communication and Race Condition
one producer thread and one consumer thread.
  • producer thread is used for writing in the shared object
  • consumer thread is used for reading from the shared object.
  • flag(like mutex) can be set to true to tell the producer, that something can be written, at that time consumer will be in blocked queue.
  • when producer thread is done, it can change the flag to False, and the producer will go to the blocked queue.
  • that is the signal for consumer to read the shared object, when the consumer is done it will change the flag to true again.

one producer thread and more than one consumer thread.
  • producer thread is used for writing in the shared object
  • consumer thread is used for reading from the shared object.
  • it will have a count that is set to 0 (count = 0) initially.
  • if count is 0 then itâ€™s producer thread turn to write something in the shared object.
  • once it is done producing, the counter will increment and the fist consumer is able to read the shared object.
  • once the first consumer finishes consuming it will increment the counter, and the second consumer is able to read the shared object.
  • these steps repeats untill the last consumer finishes consuming, then it will make count as 0 (count = 0).
  • if count is 0 then itâ€™s producer thread turn to write something in the shared object.
    Actual Race conidtion occurs when Count is not used
    • lets say there is one producer and 3 consumer initially.(obj: first producer should write something and all the cosumer should read it.)
    • first the producer will write something in the shared object and will move to the blocked queue.
    • consumer1 will come and access the shared data and it will notify all the other thread(ie. hey Iâ€™m done using it, you can come and use it )and goes to the blocked queue.
    • so the consumer1 notified before going to blocked queue, so any thread can go and access the file(even the producer can access it as well), that is called as the Race Condition, the other threads will run for accessing the file just like a race.
    • hence the objective cannot be achieved.
    • so the race condition can be avoided using count in Inter-Thread communication.
